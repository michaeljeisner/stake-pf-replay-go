# Major Refactor and Simplification Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Simplify the Stake PF Replay Desktop codebase without changing user-visible behavior, replay determinism, database contracts, generated Wails service contracts, or betting/auth safety guarantees.

**Architecture:** This is a preservation-first refactor: first freeze behavior with characterization tests and build gates, then reduce complexity at the highest-leverage seams. Treat the app as four bounded domains: Wails desktop shell, replay/backend engine, live ingest/ledger, and React frontend workflows. Refactor by extracting small pure helpers and thin adapters, not by rewriting core systems.

**Tech Stack:** Go 1.25, Wails v3 alpha, React 18, TypeScript, Vite, Vitest, SQLite, goja, modernc.org/sqlite, npm, Taskfile/Wails task runner.

  - Do **not** change SQLite schema except through migrations and migration tests.
  - Do **not** run live Stake requests during refactor validation.

## Proposed Approach

1. Establish a baseline and refactor guardrails.
2. Add characterization tests around behavior-heavy seams before moving code.
3. Split obvious mixed-responsibility files into smaller packages/modules.
4. Extract pure frontend helpers/hooks from oversized components.
5. Centralize repeated parsing/normalization/error patterns.
6. Delete compatibility cruft only after search-based proof it is unused.
7. Run targeted verification after each phase, then full verification at the end.

This is not a rewrite. The win condition is fewer responsibilities per file, smaller components, clearer seams, and unchanged outputs.

```ts
export function parseNonceInput(value: string): number | null {
  const trimmed = value.trim();
  if (!trimmed) return null;
  if (!/^\d+$/.test(trimmed)) return null;
  const parsed = Number(trimmed);
  return Number.isSafeInteger(parsed) ? parsed : null;
}

export function formatNonce(value: number | string | null | undefined): string {
  if (value === null || value === undefined || value === '') return '—';
  return String(value);
}
```

`frontend/src/lib/number-format.ts`:

```ts
export function formatMultiplier(value: number | null | undefined): string {
  if (value === null || value === undefined || !Number.isFinite(value)) return '—';
  return `${value.toFixed(2)}×`;
}

export function formatCurrencyAmount(value: number | null | undefined): string {
  if (value === null || value === undefined || !Number.isFinite(value)) return '—';
  return value.toLocaleString(undefined, { maximumFractionDigits: 8 });
}
```

## Risks / Tradeoffs

- **Risk: accidental behavior change in deterministic replay.** Mitigation: golden tests first; no RNG changes unless tests prove equivalence.
- **Risk: Wails binding drift.** Mitigation: regenerate only after service signature changes; build frontend after regeneration.
- **Risk: frontend extraction creates prop drilling.** Mitigation: first extraction keeps parent-owned state; only introduce hooks/context if repeated props become clearly worse.
- **Risk: SQL refactor changes query behavior.** Mitigation: characterize HTTP/store responses before moving query logic; constants before builders.
- **Risk: deleting dynamic uses.** Mitigation: grep symbol names and inspect string/reflection usage before deletion.
- **Risk: too much at once.** Mitigation: one seam per phase, verify after each phase, optional commit after each task with approval.

## Open Questions

1. Should the refactor optimize for desktop app maintainability first, or backend library cleanliness first?
2. Are generated `frontend/bindings/**` allowed to churn in this branch, or should service signatures remain frozen?
3. Should old docs under `docs/archive/` be cleaned or left as historical record?
4. Should frontend large-component splitting preserve all current CSS/classes exactly, or is small UI cleanup allowed?
5. Should commits be made after each task, or should all changes remain uncommitted for review?

## Recommended Execution Order

1. Phase 0 safety gates.
2. Phase 1 desktop bootstrap extraction.
3. Phase 2 live ingest/store characterization and small extractions.
4. Phase 4 frontend component splits.
5. Phase 3 backend cleanup only after frontend/live smoke remains stable.
6. Phase 5 bindings cleanup.
7. Phase 6 docs cleanup.
8. Phase 7 full verification and simplify-code review.

Do not start with dead-code deletion. That feels productive and usually summons the regression goblin with a tiny clipboard.
